  • 2.4 Constraints

    • In the case that our project will be approved by Loris, in the actual use of a school scheduling system, it would be highly important that it would be robust and fully functional. This means, no errors in the implementation of the program. Being able to cover all cases is a must. We do not want to compromise a users security. Since the users would be students, we do not want any private information leaking out. We should be aware of where we are storing the users information(eg. Cookies), and that it does not transfer over to another user logging in on the same machine for example. We want to make sure that user data is stored correctly, and retrieved correctly from the database. Most importantly, the scheduler must be easy to use and navigate through, so that it is intuitive for the unintuitive.
