3. Specific requirements

  • 3.1 External interface requirements

    • 3.1.1 User interfaces

      • 3.1.1.1 A user interface for course selection which includes:

        • 3.1.1.1.1 Overall page frame similar to current LORIS system http://telaris.wlu.ca

        • 3.1.1.1.2 A term selection dropdown

        • 3.1.1.1.3 A guided course search system with three tabs

          • 3.1.1.1.3.1 "Manual" which provides a narrowing course selection process finalizing with a description as the user clicks on each class

          • 3.1.1.1.3.2 "Course" which provides a search functionality with descriptions

          • 3.1.1.1.3.3 "CRN" which provides a system similar to the current system that can be found here.

        • 3.1.1.1.4 Two persistent buttons are located below the search systems

          • 3.1.1.1.4.1 "Add to Worksheet" which adds to the users worksheet

          • 3.1.1.1.4.2 "Register Course" which adds to the users registered courses

        • 3.1.1.1.5 A schedule view with the heading "Schedule View"

          • 3.1.1.1.5.1 A weekly calendar view

            • 3.1.1.1.5.1.1 Displays registered classes in the weekly calendar view

            • 3.1.1.1.5.1.2 Displays classes added to the "schedule viewer" in the weekly calendar view

          • 3.1.1.1.5.2 "Worksheet" which has a list of courses that the student is interested in

          • 3.1.1.1.5.3 "Schedule Viewer" which as both a list of courses that the student is registered in and courses that the student would like to preview

            • 3.1.1.1.5.3.1 Note: Classes that overlap in terms of timetable will show up a different colour on the calendar view.

    • 3.1.2 Hardware interfaces

      • The system will be located on the Wilfrid Laurier University Computer Science server Hopper. Clients will access the content from their home or university computers through the internet.

    • 3.1.3 Software interfaces

      • 3.1.3.1 At this time software requirements beyond being capable of compiling and running on hopper have not been fully discussed. The following is a list of topics covered thus far.

        • 3.1.3.1.1 Git repository at https://github.com/CP317-2013/LorisPlus

        • 3.1.3.1.2 UML created by Magic Draw

        • 3.1.3.1.3 PHP and JAVA have been discussed as possible languages for production

        • 3.1.3.1.4 Google docs and Facebook are being used for group organization as well as communication

    • 3.1.4 Communications interfaces

      • N/A

  • 3.2 System features

    • 3.2.1 Display a page

      • 3.2.1.0 Priority

        • HIGH

      • 3.2.1.1 Introduction or Purpose

        • The system shall always display a page to the user.

      • 3.2.1.2 Stimulus or Response Sequence

        • User requests from the web server the LorisPlus page and a page is always returned

      • 3.2.1.3 Associated functional requirements

        • The system shall have pages for all relevant error codes as well as function web pages

    • 3.2.2 Schedule View

      • 3.2.2.0 Priority

        • HIGH

      • 3.2.2.1 Introduction or Purpose

        • The system shall provide users with a visual display of their weekly schedule, or a potential schedule from courses placed by the user into the schedule viewer

      • 3.2.2.2 Stimulus or Response Sequence

        • The user will select to view either their current schedule or a potential one

          • If their current schedule is selected, a weekly schedule of all the user’s currently registered courses will be displayed

          • If the potential schedule is selected, a weekly schedule of all the user’s registered courses, and all courses the user has added to the schedule viewer box will be displayed

      • 3.2.2.3 Associated functional requirements

        • The system shall retrieve the user’s registered courses for the selected term from the appropriate database

        • The system shall store courses selected by the user to move to schedule viewer

        • The system shall process the details of these courses to generate a visual representation of the user’s schedule

        • The system will not modify the data of these courses when generating the display

    • 3.2.3 Class Lookup by Filter

      • 3.2.3.0 Priority

        • HIGH

      • 3.2.3.1 Introduction or Purpose

        • In order to allow users to browse possible courses to register in, the system shall provide a list of courses filtered according to the user’s selection, and provide information on each course shown

      • 3.2.3.2 Stimulus or Response Sequence

        • The user shall select which filters to apply to the courses they wish to view, the system shall then apply these filters to search through a database of courses and display a list of those which meet the criteria specified.

        • The user will then be able to select a course from the list to see detailed information, such as a description, prerequisite courses etc. along with buttons allowing them to add the course to their worksheet or register immediately

      • 3.2.3.3 Associated functional requirements

        • The system shall access a database of courses and generate a list of all which meet the user specified criteria

        • The system shall store relevant information about these courses from the database

        • The system shall provide error checks before allowing the user to register in a course (e.g. check to see if there is a schedule conflict with another course)

        • The system shall update the database should the user register in a valid course

    • 3.2.4 Class lookup by Course Name

      • 3.2.4.0 Priority

        • HIGH

      • 3.2.4.1 Introduction or Purpose

        • As users may know the course they wish to register in ahead of time, the system shall allow users to view courses by the user entering the course name into the system.

      • 3.2.4.2 Stimulus or Response Sequence

        • The user shall select the search by name tab, then enter in the course name. the system shall then retrieve this course from the database or display an error message if no such course exists

      • 3.2.4.3 Associated functional requirements

        • The system shall access a database of courses and retrieve the one whose name matches the user’s query

        • The system shall store relevant information about these courses from the database

        • The system shall display this information

        • The system shall provide error checks before allowing the user to register in a course (e.g. check to see if there is a schedule conflict with another course)

        • The system shall update the database should the user register in a valid course

    • 3.2.5 Class lookup by CRN

      • 3.2.5.0 Priority

        • LOW

      • 3.2.5.1 Introduction or Purpose

        • As users may know the course they wish to register in ahead of time, the system shall allow users to view courses by the user entering the CRN into the system.

      • 3.2.5.2 Stimulus or Response Sequence

        • The user shall select the search by CRN tab, then enter in the CRN. the system shall then retrieve this course from the database or display an error message if no such course exists

      • 3.2.5.3 Associated functional requirements

        • The system shall access a database of courses and retrieve the one whose CRN matches the user’s query

        • The system shall store relevant information about these courses from the database

        • The system shall display this information

        • The system shall provide error checks before allowing the user to register in a course (e.g. check to see if there is a schedule conflict with another course)

        • The system shall update the database should the user register in a valid course

    • 3.2.6 Remove Course

      • 3.2.6.0 Priority

        • HIGH

      • 3.2.6.1 Introduction or Purpose

        • As users may change their minds about courses they registered for, or eliminate from consideration courses on their worksheet, the system shall provide an option to deselect and deregister from courses.

      • 3.2.6.2 Stimulus or Response Sequence

        • The user shall select the button for removing/deregistering the course linked to the button. If deregistering, the system shall prompt the user for confirmation before proceeding

        • If deregistering, the system shall update the database with this information

        • If deselecting a course from the schedule viewer or worksheet, the system shall remove them from its’ own memory

        • The system will then update any relevant displays

      • 3.2.6.3 Associated functional requirements

        • The system shall provide error checks when deregistering in order to prevent invalid schedules (e.g. deregistering from a lecture but not the associated lab)

        • The system shall only access the course database when deregistering, not deselecting

    • 3.2.7 Student Login

      • 3.2.7.0 Priority

        • HIGH

      • 3.2.7.1 Introduction or Purpose

        • For the purpose of system organization and user security, each user must have an account which they will log into the system with

      • 3.2.7.2 Stimulus or Response Sequence

        • The user will enter their username into a text field, and their password into another. The system will compare this set of username and password to the one stored, and if they match, allow the user access to the rest of the system.

      • 3.2.7.3 Associated functional requirements

        • The system shall keep secure records of username password combinations

        • The system shall be able to match usernames to their respective password

    • 3.2.8 Administrator Delete existing course

      • 3.2.8.0 Priority

        • MEDIUM

      • 3.2.8.1 Introduction or Purpose

        • Removes a course from the database

      • 3.2.8.2 Stimulus or Response Sequence

        • Courses are removed with the same search feature as the user has to add a course

        • If an error occurs the system shall display an error message to administrator with no change to the database

      • 3.2.8.3 Associated functional requirements

        • The system shall remove a course if and only if no student is registered for classes

        • The system shall remove a course if and only if the course exists in the database

    • 3.2.9 Administrator Login

      • 3.2.9.0 Priority

        • HIGH

      • 3.2.9.1 Introduction or Purpose

        • Allows the Administrator to login from the same page as a user

      • 3.2.9.2 Stimulus or Response Sequence

        • See user login

        • The system shall redirect on login to Administrator pages

      • 3.2.9.3 Associated functional requirements

        • Administrators may only login if they exist.

    • 3.2.A Administrator Modify Existing Course

      • 3.2.A.0 Priority

        • MEDIUM

      • 3.2.A.1 Introduction or Purpose

        • Modifies a course in the database

      • 3.2.A.2 Stimulus or Response Sequence

        • The system shall lock the course in the database and the course cannot be registered for while the course is being modified

        • If the course is modified while a student is in the process of registering the student receives an error and the students courses do not change.

      • 3.2.A.3 Associated functional requirements

        • Only available on courses currently in the database and not during the registration process

    • 3.2.B Administrator Add New Course

      • 3.2.B.0 Priority

        • HIGH

      • 3.2.B.1 Introduction or Purpose

        • Adds a course in the database

      • 3.2.B.2 Stimulus or Response Sequence

        • The system shall prevent the addition of two of the same class with the same CRN or course code

        • If the same course or the same CRN is added the administrator is informed with an error message and no changes occur with the database.

      • 3.2.B.3 Associated functional requirements

        • Can only add new classes that are not currently represented in the database

    • 3.2.C Administrator Create User (student)

      • 3.2.C.0 Priority

        • HIGH

      • 3.2.C.1 Introduction or Purpose

        • Allows administrators to create users

      • 3.2.C.2 Stimulus or Response Sequence

        • The administrator attempts to add a new student

        • No two users may have the same ID. If an administrator attempts to add a user with the same ID an error is displayed with no change to the database

      • 3.2.C.3 Associated functional requirements

        • Administrators may only add new users

    • 3.2.D Administrator Create Administrator

      • 3.2.F.0 Priority

        • MEDIUM

      • 3.2.D.1 Introduction or Purpose

        • Allows administrators to create other administrators

      • 3.2.D.2 Stimulus or Response Sequence

        • The administrator attempts to add a new administrator

        • If two administrators exist cause error with no changes to the database

        • No two administrators may have the same ID. If an administrator attempts to add a user with the same ID an error is displayed with no change to the database

      • 3.2.D.3 Associated functional requirements

        • Administrators may only add new administrators and cannot promote users

    • 3.2.E Administrator Remove User (student)

      • 3.2.E.0 Priority

        • MEDIUM

      • 3.2.E.1 Introduction or Purpose

        • Allows administrators to remove users

      • 3.2.E.2 Stimulus or Response Sequence

        • The administrator attempts to remove an user

        • Administrators shall not remove a user that does not exist

        • If an administrator attempts to remove a user that does not exist an error is displayed and no change is made to the database

      • 3.2.E.3 Associated functional requirements

        • Users may not remove other users

    • 3.2.F Administrator Remove Administrator

      • 3.2.F.0 Priority

        • LOW

      • 3.2.F.1 Introduction or Purpose

        • Allows administrators to remove administrators

      • 3.2.F.2 Stimulus or Response Sequence

        • The administrator attempts to remove an administrator

        • Administrators may not remove themselves

        • The only administrator may not remove themselves

        • No two users may have the same ID. If an administrator attempts to add a user with the same ID an error is displayed with no change to the database

      • 3.2.F.3 Associated functional requirements

        • Administrators may only remove other administrators if there is more than one administrator.

  • 3.3 Performance requirements

    • 3.3.1 The system shall accept input from a maximum of 2 program majors at one time during the fall registration process

    • 3.3.2 The system response time shall be no longer than 1 second under optimal conditions and 5 seconds under heavy load (e.g. Fall course registration)

    • 3.3.3 The system shall lock a class in the database when added to the schedule viewer until the user logs out or until the session expires.

  • 3.4 Design constraints

    • 3.4.1 The system shall run on a modern desktop browser (e.g. Chrome 24, FireFox 24, Internet Explorer 10, etc.)

  • 3.5 Software system attributes

    • 3.5.1 Reliability

      • 3.5.1.1 The system shall maintain stability

      • 3.5.1.2 The system shall protect all course registration data from unwanted or accidental modification

    • 3.5.2 Availability

      • 3.5.2.1 Course registration shall be available during fall term in accordance with the current major registration and restriction schedule. (e.g. Computer Science 3rd year register after 4th year Computer Science students.)

    • 3.5.3 Security

      • 3.5.3.1 The user login credentials shall be salted and hashed per industry standard

      • 3.5.3.2 An user must not be able to view, change, modify other users information

      • 3.5.3.3 An user shall not be able to impersonate an administrator and vice versa

  • 3.6 Other requirements

    • 3.6.1 Must improve on the current LORIS course registration process

    • 3.6.2 Must have easily understood user interface
